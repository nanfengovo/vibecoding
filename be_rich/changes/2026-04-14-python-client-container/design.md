# Request

将现有的 Python 铜价监控脚本升级为适合容器内开发的客户端应用程序，并尽量保证容器内时间与外部宿主机保持一致。界面设计需要比普通工具页更完整，兼顾信息密度、可读性和观感。后续又追加了新的估值阈值与自动通知需求：当溢价率小于 `1.1` 时提示低估，达到 `1.3` 以上视为正常估值，达到 `1.4` 以上视为高估，并支持配置邮件、企业微信和飞书通知。最新又补充了桌面端交互要求：右侧深色说明区需要优化样式，并且通知配置必须可以直接在 UI 上编辑、保存和测试；进一步反馈指出 SMTP“安全方式”控件不清晰，盘中联动曲线的图例、刻度和双序列可读性也需要增强。

# Current Behavior

当前项目只有一个 `copper_market_monitor.py` 脚本，运行后在终端打印铜期货与 SCCO 股票信息，并计算溢价率。

当前 Dockerfile 仅用于打包并执行该脚本，缺少以下能力：

- 缺少适合持续开发的 dev container 配置
- 缺少图形化客户端界面
- 缺少统一的数据访问层，CLI 与未来客户端无法复用
- 缺少容器时区与宿主机时间语义对齐方案
- 免费公网部署后的浏览器版仍只能查看行情，无法像桌面端一样直接编辑通知、代理与测试发送

# Proposed Change

计划引入以下改动：

- 新增可复用的数据服务模块，统一处理行情抓取、时间格式化和溢价率计算
- 保留原有 CLI 监控脚本，但改为复用数据服务
- 新增基于 Qt for Python（PySide6）的桌面客户端，作为可双击启动的原生窗口
- 新增 `docker-compose.yml` 与 `.devcontainer/devcontainer.json`，支持容器内开发和直接运行客户端
- 在容器镜像内安装 `tzdata`，默认使用 `Asia/Shanghai`
- 通过挂载宿主机 `/etc/localtime` 的方式减少容器与宿主机之间的本地时间偏差
- 新增 macOS 打包脚本，产出可双击启动的 `.app`
- 调整溢价率阈值与提示文案，使低估 / 正常估值 / 高估的判断更贴近用户口径
- 新增可配置通知能力，支持 SMTP 邮件、企业微信机器人 webhook、飞书机器人 webhook
- 为通知增加冷却时间与状态去重，避免自动刷新时重复刷屏
- 在 Streamlit 浏览器版补充通知与代理配置 UI，并提供“保存配置 / 重载配置 / 测试发送”
- 为公网版配置区增加轻量管理员口令保护，避免任意访客直接改写敏感配置

# Open Questions

- 用户提到的 `stitth` 与 `uiuxpromaxskill` 不在当前可用技能列表中，暂按最接近的 `frontend-skill` 设计思路执行
- Yahoo Finance 的 `info` 接口本身存在网络与字段波动风险，需要在 UI 中容忍缺失字段
- 打包阶段需要依赖本机 Python 与 PyInstaller，首次构建会比普通脚本启动更重
- “微信”在桌面端场景下默认按企业微信机器人 webhook 实现，而不是个人微信自动化
- 用户仅明确给出了 `<1.1`、`>1.3`、`>1.4` 三个阈值，因此 `1.1-1.3` 区间需要定义成过渡观察区，避免与“正常估值”重叠

# Decision

决定采用 Qt for Python（PySide6）实现桌面客户端，原因如下：

- 可直接生成 macOS 下可双击启动的 `.app`
- 复用现有 Python 数据逻辑的成本低
- 不依赖浏览器，符合“客户端”使用预期
- 绕开系统 `Tk 8.5` 在新 macOS 上的兼容风险
- 桌面样式与绘图能力比 Tkinter 更稳定，适合工作台式界面

界面设计执行上，参考当前可用的 `frontend-skill` 思路，采用“信息工作台”而不是传统表单窗口。重点通过顶部概览、左右工作区、低噪音色彩和重点数值强调来建立层级。

时间同步方面不尝试修改容器系统时钟，而采用“共享宿主机时钟 + 显式设置时区 + 挂载 localtime”的组合方案。这种方式风险更低，也更符合容器常规实践。

通知设计方面，继续采用“桌面端 / CLI 复用统一通知服务 + 本地 JSON 配置文件”的持久化方式，但桌面端不再只暴露配置文件入口，而是增加可视化配置表单。配置仍然写入用户目录下的 JSON 文件，这样既便于打包，也能让通知配置在重新构建 `.app` 后继续保留。

桌面端右侧面板会从“长文本说明墙”调整为“估值摘要 + 关键解释变量 + 通知配置”的三段式结构。样式上强调更清楚的分组、留白和表单层级，减少一整块深色背景里混杂说明文、状态文和按钮带来的压迫感。

在 SMTP 表单里，不再继续使用两个深色复选框承载 `SSL / STARTTLS`，而改成单选式的明确控件，避免用户不知道当前到底选中了哪种加密方式。表达方式优先“SSL / STARTTLS / 不加密”三选一，减少邮件通道排查成本。

考虑到 Gmail 的 `App Password` 常按 `4-4-4-4` 分组展示，而用户往往会把带空格的字符串直接粘贴进配置，SMTP 发送前需要对密码做一次“去除所有空白字符”的归一化。这样既不改变用户视觉上的录入体验，也能显著降低 Gmail 授权失败或服务端直接断开连接的概率。

如果 Gmail SMTP 在当前网络的欢迎语或 TLS 握手阶段持续超时，桌面端错误文案需要明确指出这是网络层拦截，而不是继续让用户在账号密码上反复排查。对于 `smtp.gmail.com`，提示文案会额外建议切换到企业微信 / 飞书 webhook，或改用可访问 Gmail SMTP 的网络。

既然用户确认本机存在可用的本地代理，通知系统需要补充统一代理层，而不是继续让用户在系统外部折腾环境变量。代理设计采用“一个代理配置，复用到 SMTP 与 webhook”方式，避免邮件、企业微信、飞书分别配置不同出口。

代理配置建议支持两种来源：

- `system`：自动读取当前系统 / 环境里的代理设置，优先使用 `https`，其次 `http`
- `custom`：用户在桌面端显式填写代理类型、主机、端口和可选认证信息

代理类型优先支持 `HTTP` 与 `SOCKS5`。原因是本机常见代理工具通常会暴露 `127.0.0.1:7890` 的 HTTP 代理和 `127.0.0.1:7891` 的 SOCKS5 代理。SMTP 连接通过 HTTP CONNECT 或 SOCKS5 建立到目标邮件服务器，企业微信 / 飞书 webhook 则走同一代理配置发出 HTTPS 请求。

桌面端通知面板里需要新增一个独立的“代理设置”卡片，放在 SMTP 与 webhook 通道配置之前。表单至少包含：

- 是否启用代理
- 代理来源（系统代理 / 自定义）
- 代理类型（HTTP / SOCKS5）
- 主机
- 端口
- 用户名（可选）
- 密码（可选）

当选择系统代理时，界面仍保留字段，但以说明文字强调“当前发送将优先读取系统代理”，以降低用户对状态的不确定感。通知摘要区域也需要把“当前是否启用代理、使用哪种来源”展示出来，减少排查成本。

不过从本机实测看，双击应用或独立 Python 进程未必总能稳定读到系统代理，因此桌面端仍应优先保留“自定义代理”作为最可控方案。对于已经明确知道本地代理端口的用户，直接填写 `127.0.0.1:7890` 这类地址会比依赖系统自动发现更可靠。

针对“免费公网 + 保留桌面端”这个目标，最合适的结构不是把 Qt 桌面端硬搬到网页上，而是继续保留双形态：

- `streamlit_app.py` / `client_app.py` 负责浏览器公网版
- `desktop_app.py` 继续负责原生桌面端

浏览器版配置能力需要额外考虑“公网部署”语境。桌面端直接把表单暴露给本机用户问题不大，但 Streamlit Community Cloud 上的页面通常对外可见，因此不能默认把 SMTP 密码、webhook 和代理账户直接暴露给任何访问者。

这里采用“公共监控面板 + 管理员配置工作区”的双层结构：

- 公共区继续显示行情、溢价率、联动曲线与快照
- 管理区单独承载通知、代理和测试发送表单
- 若设置了 `admin_passcode`，则只有输入正确口令后才显示或解锁敏感字段
- 若未设置 `admin_passcode`，仍允许使用配置区，但会明确提示这在公网环境下存在安全风险

管理员口令的来源优先采用 Streamlit secrets，其次允许从环境变量读取，兼容本地与云端两种运行方式：

- `st.secrets["admin_passcode"]`
- `COPPER_PULSE_ADMIN_PASSCODE`

通知配置仍复用 `notifications.py` 的 JSON 文件持久化，保持与桌面端、CLI 的口径一致。浏览器版不另外发明一套 schema，只负责把 UI 表单映射到同一份配置结构。这样在本地运行 Streamlit 时，桌面端与网页端也能共享同一份通知配置。

不过需要接受一个现实约束：在免费公网环境里，本地文件系统持久化并不如桌面端稳定。运行中的 Streamlit 进程通常可以读写本地配置文件，但遇到平台重启或重新部署时，持久化文件可能丢失。因此 README 需要明确说明：浏览器版适合用来查看与临时调整配置；若希望长期稳定保留口令或敏感凭证，推荐借助 Streamlit secrets 管理管理员口令，并在必要时重新填写通知表单。

如果用户希望把这个项目收进自己的 GitHub 合集仓库，例如 `nanfengovo/vibecoding`，则应把当前项目作为一个子目录（例如 `be_rich/`）纳入合集，而不是依赖 GitHub 的“原生 fork 到子文件夹”能力，因为 GitHub 的 fork 语义是“仓库级复制”，不是“目录级纳入”。

盘中联动曲线会强化成“可读的监控图”而不是单纯画出两条线。重点补充以下信息层：

- 顶部图例，直接显示两条线的名称与相对基准变化
- 底部时间刻度，至少展示起点 / 中点 / 终点
- 末端标签避让，避免两条序列靠近时叠在一起
- 更稳的左右边距和底部留白，避免标签、基准线和曲线互相挤压

估值判断采用四段式：

- `< 1.1`：低估
- `1.1 - 1.3`：修复区 / 观察区
- `1.3 - 1.4`：正常估值
- `>= 1.4`：高估

通知默认只对“低估 / 正常估值 / 高估”三种明确状态发出，`1.1 - 1.3` 的过渡区默认只在界面提示，不主动推送。

桌面端 UI 配置能力按以下范围交付：

- 通知总开关由各通道独立控制
- 可在 UI 中编辑 SMTP 的主机、端口、用户名、密码、发件人、收件人及 SSL / STARTTLS
- 可在 UI 中编辑企业微信 webhook、提醒账号和提醒手机号
- 可在 UI 中编辑飞书 webhook
- 可在 UI 中编辑冷却时间与触发状态名单
- 保留“发送测试”和“重载配置”能力，便于快速验证

浏览器端 UI 会复用相同的配置能力，但布局更适合网页工作台：

- 保持顶部公开监控视图，优先保证首次进入就能看懂行情
- 将配置区放在下半部分，通过标签页或分栏组织“概览 / 通知 / 代理”
- 延续浅色工作台风格，但让配置区拥有更明确的管理态层级与操作反馈

进一步针对 Streamlit Community Cloud 的运行特征，需要处理三个额外问题：

- 每次控件交互都会触发整页重跑，如果仍然在顶层直接请求行情，会让“保存配置 / 解锁表单 / 切换标签页”都变得很慢
- 自动刷新在用户编辑长表单时会打断输入，尤其在公网环境里更明显
- 左侧 sidebar 依赖浏览器与 Streamlit 自带折叠入口，移动端和部分嵌入环境下可发现性并不好

因此浏览器版最终采用以下策略：

- 将监控配置从 sidebar 移到主工作区顶部，避免依赖默认的左上角折叠入口
- 使用 `st.cache_data` 对行情抓取做短周期缓存，并增加手动刷新 nonce，让“看盘刷新”和“表单交互”解耦
- 在通知工作区解锁时自动暂停行情自动刷新，减少输入丢失
- 使用顶层标签页拆分 `行情总览 / 通知与代理 / 计算与说明`，让长表单与图表、说明区不再挤在同一滚动流里
- 通知配置内部再用二级标签页拆分“发送策略 / 代理 / 邮件 / 企业微信 / 飞书”，减少桌面端与移动端的横向挤压

# Risks

- Qt 打包体积会比纯脚本更大
- 宿主机若不是 `Asia/Shanghai`，仅设置 `TZ` 可能与宿主机展示时区不一致，因此额外挂载 `/etc/localtime`
- macOS 下 `/etc/localtime` 为符号链接，Docker Desktop 环境通常可用，但不同机器上仍可能有路径兼容性差异
- `yfinance` 返回字段可能为空，界面需要提供降级展示
- 若本机环境变动较大，重新打包 `.app` 时可能需要再次安装构建依赖
- 邮件与 webhook 发送本身依赖网络与第三方服务配置，应用只能负责提供可配置接入能力
- 如果用户把 webhook 配置为个人微信相关第三方桥接服务，稳定性与合规性不受当前项目控制
- 自动刷新会持续触发通知判断，因此必须做好状态记忆与冷却时间控制
- UI 中会承载 webhook 和 SMTP 密码，表单既要可编辑，也要避免视觉上过度拥挤，防止误操作
- 如果用户在 UI 中输入非法 JSON 结构或空白字段，需要在保存时做最小限度的归一化处理
- 浏览器版若未配置管理员口令，就意味着任何能访问页面的人都能修改通知配置，因此必须在 UI 中清楚提示风险
- 免费公网环境的本地文件持久化能力有限，浏览器版配置在平台重启后可能需要重新填写

# Validation Plan

- 运行 Python 语法校验，确保新增与修改文件可编译
- 检查 devcontainer 配置与 compose 文件的路径是否一致
- 检查客户端主流程是否正确引用数据服务
- 确认桌面窗口能展示更新时间、当前时区、核心行情指标与溢价率
- 确认 macOS 打包命令能够产出 `.app`
- 验证新阈值在桌面端与浏览器端都显示为一致文案
- 验证缺省通知配置会自动生成且默认关闭，不会误发消息
- 验证手动测试通知和自动阈值通知都能走到配置校验与冷却逻辑
- 验证桌面端可直接编辑并保存通知配置，且保存后无需手改文件即可立即生效
- 验证右侧面板在桌面窗口下仍保持可读，不出现表单溢出或按钮拥挤
- 验证浏览器版在未配置管理员口令时仍可使用，但会明确提示公网风险
- 验证浏览器版在配置管理员口令后，未解锁状态下不会展示敏感配置字段
- 验证浏览器版可在 UI 中编辑并保存 SMTP / 企业微信 / 飞书 / 代理配置，并能调用测试发送
